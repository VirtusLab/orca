# Plan: unify autonomous and interactive backend modes

## Goal

Today the `LlmBackend` SPI carries two parallel method pairs — `runHeadless`/
`continueHeadless` for autonomous calls, `runInteractive`/`continueInteractive`
for the live ones. They share most of the work (spawn a subprocess, parse
events, build an `LlmResult`) but the autonomous pair runs claude in
`--output-format json` mode that throws away the event stream, while the
interactive pair drives a stream-json subprocess from `ClaudeConversation` /
`CodexConversation`.

Collapse both into the same conversation-shaped path with a single knob —
`canAskUser: Boolean` — that toggles the MCP `ask_user` setup. The autonomous
path becomes "open a conversation, drain it internally, return the awaited
result"; the interactive path stays "open a conversation, hand it to
`Interaction.drive`".

Side benefit: the drain helper can emit `OrcaEvent.ToolUse` /
`OrcaEvent.AssistantMessage` per `ConversationEvent`, giving autonomous calls
the per-tool-use visibility they lack today.

## Differences kept after unification

| Concern | Autonomous (canAskUser=false) | Interactive (canAskUser=true) |
| --- | --- | --- |
| MCP `ask_user` server + config | not set up | set up |
| `ask_user` hint in system prompt | omitted | appended |
| Drain ownership | inside backend (helper) | caller (`Interaction.drive`) |
| Retry policy | retries on parse fail (`LlmCall.runHeadlessWithRetry`) | no retry — user is steering |
| `displayPrompt` | `None` (no renderer) | required (renderer needs it) |
| `--json-schema` / `--output-schema` | passed when structured | passed when structured |

## Differences removed

- Claude's `--output-format json` headless path. Single output-format
  (`stream-json`) for every call. `ClaudeArgs.headless`,
  `ClaudeHeadlessResponse`, the non-piped `cli.run` invocation deleted.
- Codex's bespoke `drainHeadless` / `HeadlessAccumulator` / `foldStdout`
  re-expressed as a drain over `CodexConversation.events`. Stderr handling
  is already in `StreamConversation`.
- The autonomous structured-call path begins using `--json-schema` (claude)
  / `--output-schema` (codex initial-exec only) for CLI-side validation,
  matching what the interactive path does today. The post-hoc
  `ResponseParser` retry safety net stays.

## Tasks

### Task 1 — `OrcaEvent.AssistantMessage` + drain helper skeleton

- Add `OrcaEvent.AssistantMessage(text: String)` next to `ToolUse` / `Step`.
  Renders in `TerminalEventListener` as a `● <first line>` (truncated)
  block under the current stage indent — mirrors the assistant-prose glyph
  the live renderer already uses.
- Add a shared helper `Conversations.drainAutonomous(conv)` (in
  `orca.backend`) that walks `conv.events`, awaits result, throws on
  cancellation. Initial form is silent — no `OrcaEvent` emission yet; task 4
  wires that in. Unblocks the structural refactors in tasks 2 and 3 without
  changing user-visible output.
- Tests: extend `TerminalEventListenerTest` with rendering cases for
  `AssistantMessage` (single-line, multi-line truncation, empty).

### Task 2 — codex: `runHeadless` calls `openConversation` + drain helper

- `CodexBackend.runHeadless` / `continueHeadless` open a `CodexConversation`
  via the existing `openConversation` and call `drainAutonomous`. Initial
  `--output-schema` is preserved on the initial-exec path; resume still
  rejects the flag (codex constraint).
- Delete `invokeHeadless`, `drainHeadless`, `foldStdout`, `HeadlessAccumulator`.
- The `CodexConversation` lifecycle (cancellation on process death, stderr
  drain from `StreamConversation`) handles the edge cases the old drain did
  inline.
- Tests: existing `CodexBackendTest` headless cases pass against the new
  drain path; accumulator unit tests delete.

### Task 3 — claude: `runHeadless` calls `openConversation(canAskUser=false)` + drain helper

- Parameterise `ClaudeBackend.openConversation` on `canAskUser: Boolean`.
  When `false`: no `AskUserMcpServer.start`, no `.orca-mcp-<port>.json`, no
  `--mcp-config` arg, no `AskUserToolName` auto-approve, no `AskUserHint` in
  the system prompt.
- Rewrite `ClaudeBackend.runHeadless` / `continueHeadless` to call
  `openConversation(canAskUser = false)` + `drainAutonomous`. The
  autonomous structured path now passes `outputSchema = Some(...)` so claude
  enforces the schema CLI-side.
- Delete `ClaudeArgs.headless`, `ClaudeHeadlessResponse`, `invokeHeadless`,
  and the `includeAskUserHint = false` branch in `writeSystemPromptIfPresent`.
- Tests: `ClaudeBackendTest` headless cases stand up a fake stream-json
  subprocess; `ClaudeArgsTest` headless cases delete; `ClaudeIntegrationTest`
  runs against the unified path.

### Task 4 — drain helper emits `OrcaEvent.ToolUse` / `AssistantMessage`

- `drainAutonomous` takes an `OrcaListener` (or is passed one from the
  backend, which itself receives it from `AbstractDefaultLlmTool.events`).
- Mapping:
  - `ConversationEvent.AssistantToolCall(name, raw)` →
    `OrcaEvent.ToolUse(name, ToolInputSummary.summarise(raw, ...))`.
  - `AssistantTextDelta` / `AssistantThinkingDelta` buffered per turn; on
    `AssistantTurnEnd` emit one `OrcaEvent.AssistantMessage(buffered)` when
    non-empty.
  - `ConversationEvent.ToolResult` swallowed (would double the noise — the
    matching `→ tool: …` line already showed the call).
  - `ConversationEvent.Error(msg)` re-emitted as `OrcaEvent.Error(msg)`.
  - `ApproveTool` / `UserQuestion` would be a protocol bug on the
    autonomous path (no MCP, no approval flow). Surface as
    `OrcaEvent.Error` and continue draining — failing the read loop would
    hide subsequent events.
  - `UserMessage` swallowed — autonomous flows don't echo their own prompt
    in the log.
- Tests: stub `Conversation` feeding scripted `ConversationEvent`s into the
  drain, recording listener asserts the OrcaEvent stream.

### Task 5 — cleanup + verification

- Dead-code sweep after tasks 1-4 land.
- Smoke-test autonomous + interactive on the same agent behaviour and
  confirm the trace is identical modulo ask_user availability.
- Update docs: `README.md` "Output" glyph legend if `AssistantMessage`'s
  `●` introduces a new entry; `design.md` SPI description if it still
  references the two-path SPI.

## Edge cases / risks

- **CLI-side schema enforcement on autonomous.** `--json-schema` may reject
  responses that `ResponseParser` would have accepted (or vice versa). The
  retry safety net stays, so worst case is one extra retry. Worth a smoke
  test against the real CLI before relying.
- **`--verbose` stderr chatter.** Already filtered by `StreamConversation`
  plus the logback config; no new work expected.
- **Per-call latency.** Stream-json startup carries more flags
  (`--verbose --include-partial-messages`) and the parser iterates more
  events than the single-JSON path. Profile after task 3 to confirm no
  regression on small calls.
- **Tests that mock `runHeadless` to return a literal `LlmResult`
  (`DefaultLlmCallTest`, retry tests).** Need to mock the `Conversation`
  boundary instead. The largest test-fixture churn lives here.
- **Codex `--output-schema` on `exec resume`.** Codex rejects it. Resume
  call passes `None` (current behaviour).

## Out of scope

- Adding ask_user to codex. Capability gap stays; codex `Conversation`s
  keep `canAskUser = false`.
- Collapsing `LlmBackend.{runHeadless, runInteractive}` into a single
  `open` method. The internal unification doesn't require an SPI rename.
